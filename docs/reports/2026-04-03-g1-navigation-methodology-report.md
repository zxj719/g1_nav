# G1导航最近5天技术报告

## 范围说明

- 报告窗口：2026年3月30日至2026年4月3日
- 关注重点：方法论、问题分层、失败模式、混杂因素识别、控制变量方式、策略组合
- 不作为重点：具体代码实现细节
- 补充说明：2026年3月27日的 probability map frontier 设计作为背景参考，但不计入本次“最近5天”主分析窗口
- 补充说明：2026年4月6日工作区里存在一轮更激进的二次收缩调参尝试，本报告不将其视为已验证结论

## 一、阶段结论

截至2026年4月3日，G1 仍然无法在完全无人干预的条件下稳定完成一次自主建图，但系统已经从“失败现象混杂、难以定位根因”的阶段，进入“问题分层清楚、可按模块控制变量”的阶段。最近5天最重要的进展，不是单点参数优化，而是把导航失败拆成了四类问题：

1. 地图前端质量问题
2. frontier 目标生成与更新问题
3. Nav2 执行阶段的贴障失败问题
4. 运行时恢复与状态管理问题

这一步拆分非常关键，因为3月30日测试中表现出来的“卡死、误判、反复去同一点、导航终止”，表面相似，根因却并不相同。如果不先分层，任何调参都会互相污染。

## 二、问题分层与混杂因素识别

### 1. 不属于 frontier 或 Nav2 参数本身的问题

这几类问题在现场表现为“导航失败”，但本质上不是 frontier 选点算法或 inflation 参数单独能解决的：

- 3D LiDAR 前端容易打到地面，产生大量燥点，导致路径封闭、建图提前结束。这是建图前端质量问题。
- 行人突然站到机器人前方导致卡死。这是动态避障缺失问题，不是静态规划参数问题。
- 转向时或行进中被误判在障碍物内部。这更像 SLAM 漂移和规划/代价地图不一致，需要脱困机制。
- 电梯材质导致地图无法封闭，frontier 持续被吸引到电梯门。这是环境感知耦合问题，需要额外近场感知补充。
- 通道尽头转向时间长，和 CycloneDDS、共享内存、底层通信链路有关，属于底层时延和控制链路问题。

### 2. 最近5天重点聚焦的问题

在剥离掉上述混杂因素后，最近5天真正聚焦并持续推进的是两条主线：

- frontier 选点与更新机制
- Nav2 代价地图与执行稳定性调参

这两条主线的共同方法论是：先把失败模式拆开，再只改一个层次，避免一次同时修改 frontier、controller、BT 和感知链路。

## 三、模块一：Frontier选点算法与更新机制

### 1. 原始问题

3月30日到4月3日这条主线集中暴露了四个问题：

1. 由于 SLAM 墙点在 grid map 中不连续，frontier 会生成在墙体内部
2. 一个 frontier 经过吸附后，原始位置还会再次生成 frontier，形成二次干扰
3. 不可达 frontier 仍然会反复尝试前往
4. 已拉黑 frontier 会重新出现，而原本可达 frontier 反而不恢复

### 2. 关键识别：frontier身份与导航目标被混在一起

最近5天最重要的认知进展，是识别出一个顶层设计错误：系统把 frontier 的“身份”和最终发给 Nav2 的“导航目标”当成了同一个点。

这会带来两个直接后果：

- goal 被吸附到旁边后，原始 frontier 又会被当成“新 frontier”重新发现
- blacklist、完成判定、preempt 比较都围绕吸附后的点，导致状态语义混乱

因此，后续策略不再围绕“继续修补吸附半径”，而是先把身份语义拆开：

- `frontier_anchor`：表示原始 frontier 质心，只负责 frontier 身份
- `nav_goal`：表示真正发给 Nav2 的导航落点，只负责执行目标

这是最近5天 frontier 策略设计的核心转折点。

### 3. 控制变量方式

在这一轮里，没有先去改 planner 或 controller，而是只处理 frontier 这一层的目标生成逻辑，目的是单独回答两个问题：

- frontier 是否应该被认为是“存在且可选”的
- 这个 frontier 是否存在一个当前可导航的落脚点

也就是说，frontier 发现和最终 goal 选择被明确拆成两个阶段：

- frontier 发现基于 `grid_map`
- 最终 nav goal 准入基于 `grid_map` 与 `global_costmap` 的联合判定

### 4. 已验证策略

最近5天在 frontier 层形成的策略组合如下：

- 每次准备发新 frontier goal 前，必须基于最近一次收到的 `global_costmap` 重新判定，而不是“等到过一次就一直沿用”
- 废除增量式的旧 frontier 更新思路，改为更实时、全局的 frontier 重算，以适配 SLAM 后端优化和单子图模式
- 废除“不断向外扩张”的 goal 搜索半径机制，改成固定半径的圆环吸附
- 将 goal 吸附半径与 frontier 非极大值抑制半径统一，避免同一片区域重复产生多个本质相同的 frontier
- 若 frontier 自身不满足落脚条件，则只在固定圆环上寻找可行目标；圆环上没有合格目标则直接拉黑
- 黑名单不再依赖简单的时间超时，因为时间超时会让坏点反复回来干扰导航
- 当系统发现没有任何 active frontier 可用时，再整体清一次黑名单，允许一次全局重试

### 5. 方法论意义

这一轮 frontier 设计的意义，不是“把某个点吸附得更好”，而是把 frontier 系统从“启发式补丁集合”提升到“有明确身份、准入条件和恢复策略的状态机”。

它直接回应了3月30日和4月3日之间的两个核心症状：

- frontier 生成在墙体里
- 不可达点与已拉黑点持续干扰导航

## 四、模块二：Nav调参与代价地图策略

### 1. 原始问题

典型现象是：全局路径在 RViz 中看起来可接受，但机器人靠近障碍物后会抖动、停顿、转向，最后出现 `Navigation aborted by Nav2`。这一现象具有概率性，在狭窄空间中更容易触发。

### 2. 混杂因素识别

这类问题不能简单归结为“planner 不行”或“controller 不行”，因为里面混有三层因素：

- global costmap 决定全局路径是否过于贴边
- local costmap 与 controller critics 决定局部执行是否过于敏感
- progress timeout、embedded 判定、SLAM 漂移会放大贴障时的不稳定

最近5天采用的策略不是全面联调，而是先只调整 inflation，尽量不同时动 controller、BT 和恢复树。这样才能判断“贴障 abort”是否主要来自代价场几何设置。

### 3. 控制变量方式

这一轮 Nav 调参采用的是最小变量策略：

- 不先动 controller critic 权重
- 不先动 BT 恢复逻辑
- 不先动 progress checker 阈值
- 先只改 global 和 local inflation 的关系

这样做的目的，是把“路线太贴边”和“执行太神经质”拆开看。

### 4. 已验证策略

最近5天 Nav 调参形成的策略是：

- 将机器人 footprint 统一收敛为圆形语义
- global costmap 的 inflation 更保守，使全局路径更早绕开障碍
- local costmap 的 inflation 略小于 global，使局部执行不要在贴边时过早判定失败

这套方法论背后的分工非常明确：

- global 层负责让路径不要过于激进
- local 层负责让执行不要过于僵硬

它并不承诺彻底消除 abort，但能显著降低“全局看起来能走，局部却在障碍边缘抖动并放弃”的概率。

### 5. 方法论意义

这里的关键不是某个具体数值本身，而是确认了一个长期可复用的原则：

全局规划与局部执行不应承担相同的保守度。两者职责不同，代价地图也不应完全同构。

## 五、模块三：运行时恢复与结果判定策略

最近5天另一个重要变化，是开始显式区分“真正不可达”和“切目标过程中的中间失败”。

过去系统更接近这样的逻辑：

- 只要 abort，就拉黑当前 frontier

这种处理方式太粗暴，会把“handoff 过程中的 abort”和“环境真的不可达”混在一起。最近5天逐步形成的新策略是：

- 真正的 goal rejected 或稳定 abort，才视为 frontier 不可达
- 若 abort 发生在目标切换 handoff 过程中，不立即拉黑当前 frontier
- 若所有 frontier 都被筛掉，再清空黑名单做一次全局重试

这说明系统已经开始从“单条规则驱动”转向“基于运行时上下文的结果解释”。这对现场稳定性很重要，因为导航系统里有很多失败其实是状态切换副作用，而不是真正的环境不可达。

## 六、已验证的策略组合

截至2026年4月3日，可以认为已经形成了下面这组可组合策略：

### 1. frontier层

- frontier 只负责表示未知边界，不直接等同于导航落点
- frontier 更新必须适配全局地图重排，不能依赖旧的增量缓存

### 2. 目标合法性层

- 最终落点要同时满足地图语义和代价地图语义
- 不满足时只在固定圆环内寻找替代点，不做无限扩张式搜索

### 3. 去重抑制层

- 用与吸附半径一致的 NMS 半径抑制重复 frontier
- 防止同一区域以“重新生成 frontier”的形式反复干扰导航

### 4. costmap层

- global 更保守，local 略放松
- 不用同一套代价场要求同时满足全局路径质量和局部执行稳定性

### 5. 恢复层

- 失败结果要按上下文解释
- 黑名单恢复采用“无 frontier 时整体恢复”而不是“时间到自动放行”

## 七、当前仍未闭环的问题

截至本报告窗口结束，以下问题仍然未完成闭环：

1. 3D LiDAR 地面点带来的地图燥点问题
2. 行人等动态障碍导致的局部卡死问题
3. 机器人被误判进入障碍物后的脱困机制
4. 电梯门等特殊材质环境下的建图封闭问题
5. 建图完成后自动返回初始点功能
6. 底层通信时延对转向和窄空间调头的影响

这些问题说明，当前系统虽然已经具备较清晰的方法论框架，但距离“完全无人工干预的稳定自主建图”仍有明显差距。

## 八、总结

最近5天最重要的成果，不是单个参数或单个 patch，而是方法论上的三点收敛：

1. 先分层，再调参
2. 先识别混杂因素，再解释失败
3. 先验证单模块策略，再组合成运行时方案

对于 G1 导航而言，这意味着系统已经从“问题很多但难以归因”的状态，进入“知道哪些问题属于 frontier，哪些属于 Nav2，哪些属于地图前端，哪些属于恢复策略”的状态。

这一步本身就是工程上的显著进展，因为只有在边界清楚之后，后续的避障、脱困、地图质量优化和更激进的现场调参才有可能真正形成闭环。
